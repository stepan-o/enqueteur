# 🌐+🌿 LOOPFORGE VIEWER STRATEGY SPEC
## _Web + Godot Combo (Debug → Isometric) with Unreal as Cinematic Stage_

**Status:** Draft v1.0  
**Applies to:** Sim5 → SimX (Kernel-first architecture)  
**Primary Objective:** Build a protocol-native viewer stack that scales  
**Non-Objective:** Build simulation logic into any viewer

---

## 0. Decision

We will pursue a **Web + Godot viewer combo**, with:

1. **Web Viewer** first
    - begins as a debug truth machine
    - evolves into a “watch the city evolve” isometric viewer

2. **Godot Viewer** second
    - becomes the aesthetic 2.5D/isometric experience

3. **Unreal Viewer** third
    - cinematic escalation layer once KVP stability is proven

---

## 1. Architectural Law

Loopforge is a **headless deterministic kernel**.

SimX (Rust) runs:
- locally or remotely
- deterministically
- authoritatively

Viewers run:
- as clients
- as renderers
- as input emitters

The relationship is defined entirely by:

**KVP-0001 (Kernel ↔ Viewer Protocol)**

If an interaction is not expressible via:
- `FULL_SNAPSHOT`
- `FRAME_DIFF`
- `INPUT_COMMAND`

…it does not exist.

---

## 2. System Topology

```text
SimX Kernel (Rust, headless)
        ⇄  KVP-0001 (WS/HTTP)
Web Viewer (Debug → Isometric)  +  Godot Viewer (Aesthetic Isometric)
        ⇄  (future)
Unreal Viewer (Cinematic Adapter)
```

Key property:
Kernel never imports viewer code
Viewer never imports kernel code
State truth lives only in the kernel

---

## 3. Why Web First

### Web Viewer Phase 1 = Debug Truth Machine

Web is the fastest path to:
- protocol compliance validation
- replay fidelity testing
- golden-trace inspection UI
- rapid iteration on diff semantics

This is not a toy UI.
It is the **protocol compliance harness** for the entire program.

---

## 4. Web Viewer Phase 1 — Minimum Features

Web Viewer must implement:

- KVP connect + handshake
- snapshot fetch + apply
- diff stream apply
- tick timeline (scrub/seek)
- inspector panels (agents/rooms/events)
- step_hash parity display
- desync banner + recovery action
- optional debug probes

Primary output:
- the fastest possible “truth lens” into the kernel.

---

## 5. Web Viewer Phase 2 — “Watch the City Evolve”

Once phase 1 is stable, the Web Viewer evolves into:

- 2.5D/isometric rendering
- agents as sprites / markers
- speech bubbles + inner monologue overlays
- psycho-topology overlays (tension/gossip/curiosity)
- camera pan/zoom
- time controls: pause / play / speed / seek

Web becomes a real viewer experience,
not just tooling.

---

## 6. Why Godot Second

Godot becomes the **aesthetic viewer**.

Godot is best used after Web proves:
- KVP semantics are stable
- replay + golden traces are correct
- snapshot/diff schemas are not thrashing

Godot then focuses on:
- Disco Elysium vibe
- layout polish
- animation + transitions
- “city watching” atmosphere

---

## 7. Godot Viewer Requirements

Godot Viewer must implement the same contract as Web:

- KVP connect + handshake
- apply FULL_SNAPSHOT baseline
- apply FRAME_DIFF incrementally
- render narrative fragments
- support replay seek + scrub
- handle desync recovery

Godot may add:
- higher fidelity composition
- richer presentation layers
- stronger “feel”

---

## 8. Why Unreal Third

Unreal is the **cinematic escalation layer**.

It is expensive to iterate early.
It pays off massively once the protocol is stable.

Unreal is for:
- trailers
- demos
- walking the city
- cinematic cameras + lighting
- “holy shit” presentation

Unreal is not a correctness layer.
Correctness must already be solved.

---

## 9. Viewer Requirements Comparison

### Web Viewer
- fastest iteration
- best debug + inspection UI
- easiest replay tooling
- distribution-friendly

### Godot Viewer
- best for 2.5D/isometric vibe
- best for DE-style UI + bubbles
- strong custom tooling potential

### Unreal Viewer
- best cinematic visuals
- best cameras/lighting/VFX
- heaviest integration cost

---

## 10. What SimX “Runs On”

SimX runs on:
- **Rust headless kernel**
- local machine or remote server

SimX does NOT run “inside” any viewer.

Viewers connect as clients.
This is required for:
- engine independence
- multi-viewer support
- future distribution shells (UEFN-like)
- protocol stability

---

## 11. Realistic Learning Curves

### Web Viewer
- 1–2 weeks: debug viewer baseline (connect + snapshot + diff)
- 3–6 weeks: replay scrubber + inspectors + overlays
- 2–3 months: polished isometric “watch” experience

### Godot Viewer
- 1–2 weeks: connect + render baseline
- 4–8 weeks: high-quality isometric viewer + bubbles
- 2–4 months: production vibe + tools

### Unreal Viewer (later)
- 6–10 weeks: solid viewer adapter (if KVP is stable)
- 3–6 months: cinematic quality + pipelines

---

## 12. Dead Ends to Avoid

Do NOT invest in:
- engine replication systems (KVP is replication)
- viewer-side simulation shortcuts
- viewer-side authoritative memory
- “temporary hacks” that bypass protocol
- deep engine-specific workflow automation before KVP stabilizes

These create drift and lock-in.
They do not translate.

---

## 13. Transferable Skills (Invest Here)

High-value, long-lived investments:

- protocol discipline + schema versioning
- canonical ordering + float quantization
- snapshot rebuild + diff apply patterns
- replay pipelines + golden traces
- desync detection + recovery UX
- visualization overlays (heatmaps, timelines)
- stable IDs + deterministic system ordering

These survive any viewer engine.

---

## 14. Web Tech Lane Recommendation

Web viewer should prioritize 2D isometric efficiency.

Preferred lanes:
- **Canvas 2D** (fastest shipping, simplest)
- **PixiJS** (best 2D performance + ergonomics)

Avoid for now:
- full Three.js 3D (overkill unless needed)

Decision rule:
- if overlays + sprites dominate → Pixi/Canvas
- if cinematic 3D dominates → that’s Unreal later
