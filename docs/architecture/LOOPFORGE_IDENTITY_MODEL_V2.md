Loopforge Episode Identity Spec
“Nicest Million Dollars in the Future” Edition
0. Intent

Stabilize episode identity across:

Simulation logs

Run registry

Analysis

CLI tools

Stage API

…so that:

/episodes/latest and /episodes/{id} behave predictably,

we never silently “lie” about which data an episode is built from,

future multi-run / multi-episode-per-run features can slot in without rewrites.

This spec does not prescribe every small refactor; it defines invariants and responsibilities that future code must respect.

1. Core Principles

Logs Are Canonical Identity

logs/loopforge_actions.jsonl is the source of truth for:

run_id

episode_id

episode_index

If these are missing or inconsistent in logs, the system must treat that episode identity as invalid (no guessing).

Registry Mirrors Logs (Never Invents)

EpisodeRecord entries in logs/loopforge_run_registry.jsonl must only use IDs that actually appear in the action log.

The registry can add metadata (created_at, days, steps_per_day, flags), but it cannot be the first place where an ID exists.

Analysis Is Strict and Honest

analyze_episode(...) and analyze_episode_from_record(...):

Require that all rows used for analysis have consistent (run_id, episode_id, episode_index).

Must reject (with a clear error) when there is no matching log data.

APIs Are Predictable, Not Magical

The Stage API (/episodes, /episodes/latest, /episodes/{episode_id}) never silently falls back to “some other episode.”

If identity is broken → callers get:

a clear HTTP status (404 / 409 / 422 / 424 depending on choice),

and a diagnostic JSON body.

CLI UX Is Helpful but Cannot Violate Identity

CLI conveniences (e.g. view-episode with no IDs) can infer “latest” from logs, but:

they must not write registry entries with IDs that don’t exist in logs,

they must not backfill logs with synthetic IDs.

2. Data Model & Schema
   2.1. Action Logs (Canonical)

Each ActionLogEntry row must eventually satisfy:

{
"run_id": "06d0ca368b2e4567a54c939a39191f18",
"episode_id": "06d0ca36-0000-9ffad0fa",
"episode_index": 0,
"day_index": 0,
"step": 17,
"agent_name": "…",
"role": "…",
"intent": "…",
"plan": { ... },
"perception": { ... },
"outcome": { ... }
}


Invariants:

For a given (run_id, episode_id, episode_index):

All rows for that episode share those values.

There may be multiple runs in the same file; selection is always by (run_id, episode_id) (and optionally episode_index when >1 episode per run appears).

2.2. Episode Registry

loopforge.analytics.run_registry.EpisodeRecord (conceptual shape):

@dataclass
class EpisodeRecord:
run_id: str
episode_id: str
episode_index: int
created_at: str
steps_per_day: int
days: int
# Optional, future-proof fields
status: str | None = None  # "resolved" | "unresolved" | "orphaned" | "synthetic" (see below)
source: str | None = None  # "simulation" | "cli-view-episode" | "import" | ...


Million-dollar rule:

An EpisodeRecord must only be “resolved” if there is at least one log row where:

row.run_id == record.run_id

row.episode_id == record.episode_id

row.episode_index == record.episode_index

Status semantics:

resolved
→ IDs exist in logs; analysis can safely run.

orphaned
→ registry ID does not match any log rows; analysis must refuse or surface a special error.

unresolved (optional)
→ CLI or tooling created a record before verifying logs; should be rare and ideally eliminated over time.

synthetic (optional tag in status or source)
→ IDs were generated by a tool (e.g. CLI) but never written to logs; such records should not be used for analysis.

We don’t have to implement all flags on day one, but the schema should allow them.

3. Component Responsibilities
   3.1. Simulation (run_simulation & logging)

Goal: Always write coherent, identity-bearing logs.

Invariants:

At simulation start:

Generate run_id = generate_run_id()

Set episode_index (currently 0, but future-safe).

Generate episode_id = generate_episode_id(run_id, episode_index)

Every call to log_action_step(...) must pass:

run_id

episode_id

episode_index

and the current day_index & step.

LLM path:

If a “legacy LLM” or alternative decision path exists that currently bypasses logging or identity:

Either:

(a) wire it into log_action_step with the same IDs, or

(b) explicitly disallow registry + analysis for that mode (no EpisodeRecord, no /episodes/latest coverage).

3.2. Logging Utilities (loopforge.core.logging_utils)

log_action_step(...):

Already merges identity_dict(...) into the action entry.

Future guardrail: consider a debug/assert path (behind env flag) that warns if identity fields are missing when we are in a mode that expects episodes.

read_action_log_entries(...) & helpers:

Must preserve all identity fields untouched.

Helper to be added (see §4): detect_latest_episode_identity(action_log_path).

3.3. Analysis API (loopforge.analytics.analysis_api)

analyze_episode(...):

Keep strict filtering by (run_id, episode_id):

If any rows lack these fields → raise ValueError (this signals a serious data issue).

If no rows match the requested identity → raise ValueError with clear message.

analyze_episode_from_record(record, ...):

Treat record as an intent: “analyze this episode identity.”
It must not silently relax constraints.

If we later add status, this function may:

Reject non-resolved records early, or

Allow them only under explicit “developer mode”.

3.4. Registry (loopforge.analytics.run_registry)

Responsibilities:

Provide append-only episode metadata.

Ensure appended entries are either:

Verified resolved, or

Clearly marked as orphaned/unresolved.

New helper (conceptual):

def verify_episode_identity_in_log(
action_log_path: Path,
run_id: str,
episode_id: str,
episode_index: int,
) -> bool:
"""Return True if at least one log row matches this identity."""


append_episode_record(record) can optionally:

Check this helper; if it returns False, either:

Do not write the record, or

Write it with status="orphaned", depending on policy choice.

4. CLI Behavior (view-episode, replay-episode, etc.)
   4.1. view-episode Identity Resolution

Current problematic behavior:

When run_id / episode_id are None, it generates new IDs and appends an EpisodeRecord with them — which do not exist in logs.

Million-dollar behavior:

When called without explicit IDs:

Attempt to detect latest identity from logs.

Add a helper in analysis or logging module:

def detect_latest_episode_identity(action_log_path: Path) -> tuple[str, str, int] | None:
"""
Scan the action log and return (run_id, episode_id, episode_index) for the
most recent episode, or None if no usable identity is found.
"""


Rough behavior:

Read logs in streaming mode (tail-first or scan all).

Identify the last contiguous block of entries sharing the same (run_id, episode_id, episode_index).

Return those IDs.

If detection succeeds:

Use those IDs in:

summarize_episode(...)

EpisodeRecord(...)

Registry entry is then guaranteed to correspond to existing logs.

If detection fails (e.g. log empty or truly lacks IDs):

Do not generate synthetic IDs for registry.

Either:

show a helpful CLI message:

“Could not determine episode identity from logs; no registry entry was written. You can still view this summary, but it won’t appear in /episodes.”

and either:

(a) still produce a summary without appending to registry, or

(b) fail with a clear error and require the user to rerun the sim.

In either case: no registry write with synthetic IDs.

When IDs are provided explicitly:

Keep current behavior:

Use given IDs.

Append EpisodeRecord only after verifying they appear in logs (via verify_episode_identity_in_log).

4.2. replay-episode, export-stage-episode, etc.

These should assume that registry + logs are consistent.

If in practice they encounter an orphaned record:

Output a clear error:

“Episode identity not found in action logs (run_id=…, episode_id=…). Registry entry appears orphaned.”

They should never invent new IDs.

5. Stage API Behavior

Endpoints:

GET /health

unchanged.

GET /episodes

Currently returns minimal record fields.

Million-dollar extension (backwards compatible):

Add optional fields:

status (if present)

source (if present)

GET /episodes/latest

Selection rule:

Load all EpisodeRecords.

Prefer most recent resolved record (by created_at, then episode_index).

If only orphaned/unresolved records exist:

either:

return 404 with:

{
"detail": "No resolved episodes available",
"code": "no_resolved_episode"
}


or return 424 Failed Dependency with a similar body.

Analysis rule:

Call analyze_episode_from_record with that record.

If analysis raises ValueError due to mismatch, the API should:

Catch it,

Map to a structured error, e.g.:

{
"detail": "Episode identity not found in action log",
"code": "orphaned_registry_record",
"run_id": "...",
"episode_id": "..."
}


No raw tracebacks to users.

GET /episodes/{episode_id}

Similar behavior, but:

Selection is by episode_id (maybe multiple records exist; pick latest resolved one).

Same structured error if orphaned.

6. Migration & Implementation Plan
   Phase 1 — Guardrails & Helpers (No Behavioral Change Yet)

Add:

detect_latest_episode_identity(action_log_path).

verify_episode_identity_in_log(action_log_path, run_id, episode_id, episode_index).

Add unit tests for both helpers with small in-memory JSONL fixtures.

Phase 2 — CLI Identity Discipline

Update view-episode:

For no-ID calls:

Use detect_latest_episode_identity.

If it fails, do not append registry and print a clear message.

For explicit IDs:

Use verify_episode_identity_in_log.

If false, either:

refuse to append registry, and/or

warn the user.

Add tests:

“Sim → view-episode → registry” happy path:

Run a small sim, call view-episode --latest or without IDs, then assert:

logs and registry IDs match.

“No logs” / “broken logs” cases:

Assert that view-episode does not write registry records with synthetic IDs.

Phase 3 — Registry Flags (Optional but Future-Proof)

Extend EpisodeRecord to include status and source.

Update registry read/write logic to handle missing fields gracefully (back-compat).

When appending:

Set status="resolved" when verify_episode_identity_in_log returns true.

If we choose to still write records when it returns false:

mark them as status="orphaned".

Update tests to cover both resolved and orphaned example records.

Phase 4 — API Hardening

Update /episodes to include status/source when available.

Update /episodes/latest and /episodes/{episode_id}:

Select only resolved records for normal behavior.

Catch ValueError from analysis and translate to structured HTTP error.

Add API tests:

“happy path” with coherent logs + registry → 200.

“orphaned registry record” → 404/424 with { code: "orphaned_registry_record" }.

Phase 5 — Cleanup & Documentation

Update docs:

docs/episode_identity_spec.md (this file).

docs/API.md section on /episodes and /episodes/latest.

Developer docs / CONTRIBUTING:

“How to reset local state and get a clean resolved episode.”

“What it means if you see orphaned_registry_record.”

Optionally add a make target:

make run-latest-episode:

wraps:

uv run loopforge-sim --no-db --steps ...

uv run loopforge-sim view-episode --latest

uv run loopforge-sim api-server

purely as a developer convenience.

7. Open Questions for Future Phases

Multi-episode per run:

How do we want to group logs? (Currently episode_index is always 0.)

Do we ever want /episodes/latest per-run, not global?

Scalability:

When logs become huge, do we shard per run or per day?

Should registry include a pointer to a specific log file (e.g., log_path)?

User-facing semantics:

Should /episodes eventually expose more than metadata (e.g. partial summary, tags, scenario names, etc.)?
